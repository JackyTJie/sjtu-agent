### 开发方式

## 一、安全性（合作的前提）

作为学生项目，我们清楚安全是合作的前提。以下措施已在代码中实施并持续维护。

**已实施的安全措施：**

- 用户凭据储存在本地文件系统，**不上传任何远程服务器**
- 凭据文件自动设置 `0o600` 权限（Unix/macOS）
- 飞书 Bot 支持白名单访问控制（`feishu_allowed_open_ids`）
- `/hw` `/news` `/eat` 等所有命令均需通过白名单校验
- Python 代码执行时自动剥离 `JACCOUNT_PASSWORD`、`ZHIYUAN_API_KEY` 等敏感环境变量
- 外部 HTTP 请求强制 HTTPS，URL 校验阻止 SSRF（私有 IP、回环地址）
- 文件路径 `resolve() + is_relative_to()` 防路径穿越
- 飞书消息 ID 去重，防重放攻击
- `atomic_write_json` 保证配置文件写入原子性（崩溃不丢数据）

## 二、为什么需要官方支持

作为学生项目，我们**打通了所有能通过公开渠道获取的数据源**。但以下能力，学生力量做不到：

| 重要度 | 想做但做不了的事 | 卡点 | 需要的支持 |
|--------|-----------------|------|-----------|
| ★★★ | 食堂推荐接入消费记录 | 一码通 API 加密封闭，仅限交我办 | 脱敏消费数据接口 |
| ★★★ | 校历自动获取（节假日/调休） | 每年手动从党政办通知录入 JSON | 校历 ICS 订阅或 API |
| ★★★ | 稳定的 jAccount 认证 | 目前靠 Playwright 模拟浏览器登录 | jAccount OAuth 接入 |
| ★★☆ | 课表/成绩数据稳定获取 | 教学信息服务网爬虫偶发失败 | 教务系统数据接口 |
| ★★☆ | 降低部署门槛 | 学生没有公网服务器和域名 | 学校云计算资源托管，可选模块化安装 |
| ★☆☆ | 移动端入口 | 独立 App 推广成本高 | 接入"交我办"小程序 |

每一行背后都是一个只差"最后一公里"的功能。

### 如果接入交我办，大模型怎么办？

这是一个需要坦诚讨论的问题。目前项目依赖外部 API（致远一号、DeepSeek、OpenAI 等兼容接口），用户自行配置 Key。如果接入学校官方平台，有几种可能的方案：

1. **致远一号**：交大已有的免费 LLM 平台，学生用 jAccount 即可使用。如果学校能提供更稳定的 API 配额，是最直接的方案。
2. **按量付费、学生自选**：平台托管 Bot 框架，LLM 调用走学生自己的 API Key，费用透明，自主可控。
3. **学校统一采购**：如果信息化办公室有意愿，可为全校学生提供基础额度的 LLM 调用（类似致远一号当前的模式，但更稳定）。

我们倾向于方案 1 或 2——**不对学校产生额外经费压力，也不替学生做选择**。

### 关于部署：希望支持模块化安装

目前项目是一个整体安装包。但不同同学的需求差异很大——有人只想要飞书 Bot，有人只需要 DDL 聚合，有人想用食堂推荐但不关心作业功能。

如果能接入学校云计算平台，我们希望能实现**模块化安装**：同学在平台上选择自己需要的功能（如"食堂推荐 + 课表""DDL + 新闻"），系统自动拉起对应的服务，而不是一股脑全装上。这样既降低资源占用，也让新手不会被复杂的功能列表吓到。

### 已接入的校园数据源

| 数据源 | 获取方式 | 对应功能 |
|--------|---------|---------|
| Canvas LMS | API（需用户手动获取 Token） | 作业下载/分析、文件浏览、DDL |
| 教学信息服务网 | 模拟登录 + 爬虫 | 课表查询、成绩查询 |
| 物理实验平台 phycai | 模拟登录 + 爬虫 | 实验时间、地点 |
| 中国大学 MOOC / AI 好课 | 模拟登录 + 爬虫 | DDL 聚合 |
| campuslife API | **公开接口（无需认证）** | 食堂实时拥挤度 |
| 教务处通知 | 网页抓取 | 校园新闻聚合 |
| 水源社区 | 网页抓取 | 热帖分析 |
| 交大邮箱 | IMAP（readonly） | 新邮件飞书推送 |

### 飞书 Bot 主要命令

```
/eat              实时拥挤度 + 个性化食堂推荐 + 偏好学习
/news             校园新闻 AI 摘要（教务处+水源+交大新闻网+Canvas）
/hw do 3          下载并分析第 3 个作业（Canvas → LLM → 解答）
/hw due 7         查看 7 天内截止的 DDL（四平台聚合）
/template bachelor-thesis   一键套用 SJTU 毕业论文 LaTeX 模板
/aihot            今日 AI 圈精选新闻
```

### `/eat` 是怎么做出来的

食堂推荐功能来自 @JackyTJie 的贡献（PR #91，885 行代码 + 81 个测试）。能实现是因为 **campuslife API 恰好是公开的**。

但推荐精度可以更高——如果能接入一码通消费记录（脱敏），系统就能学习用户的真实口味和用餐习惯。@JackyTJie 曾尝试抓包交我办：

> "全都加密了，认证非常严格，仅限于交我办软件访问。除非先获得官方支持，不然做不出来。"

这个例子很能说明问题：学生能把公开接口用到极致，但封闭接口才是真正提升体验的关键。

## 五、我们的底线

做这个项目，从一开始就有几条不会让步的原则：

1. **目标是方便同学，不是妨碍或越界**。我们不做考勤监控、不做行为分析、不推送广告。所有功能都以"帮同学省时间"为唯一标准——查 DDL 比自己翻 Canvas 快，看食堂拥挤度比自己跑去窗口快，收新闻摘要比自己翻十几个网页快。

2. **保障知情权，但保留完全主动权**。Bot 会告诉同学"今天有 3 个 DDL 截止"——但不会替同学做任何决定。屏蔽某类新闻（`/news_block`）、重置推荐画像（`/news_reset`）、关闭日报推送——所有选项都掌握在学生自己手里。我们提供信息，学生做选择。

3. **数据主权在学生**。当前所有用户数据（对话记录、用餐偏好、课表缓存）存储在本地。如果未来接入学校平台，我们同样坚持：学生可以随时导出自己的数据，可以一键删除所有历史记录，可以选择不参与任何数据收集。
